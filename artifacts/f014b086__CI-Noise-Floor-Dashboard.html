Overview

TL;DR
Lower jitter means smaller regressions become detectable. Bare‑metal and tuned dedicated EC2 host the tightest noise floors; GitHub‑hosted runners, QEMU emulation, and CPU‑quota‑capped containers should be avoided for regression‑sensitive benchmarks.

Methodology & sources

How the benchmarks and the noise model were assembled.

Workloads

  • Matrix multiply — naive triple‑loop 512×512 f64 in C. ALU‑bound, cache‑friendly.
  • Prime sieve — Sieve of Eratosthenes up to 107. Memory‑bound with long linear scans.
  • Each workload runs 200 times per environment with a warm‑up pass discarded.

Measurements

  • Wall‑clock — CLOCK_MONOTONIC_RAW around the kernel, resolution < 1 µs.
  • Instruction count — valgrind --tool=cachegrind counting retired instructions (Ir).
  • Jitter — coefficient of variation (σ/µ), p99/median tail ratio, IQR.

Noise-floor priors

  • GitHub‑hosted runners measured at 2.66% CV over 100 runs — CodSpeed, 2025.
  • Tuned EC2 with C‑states off, HT off drops variance from 20–30% to <5% — MongoDB / DEV, 2026.
  • QEMU user‑mode TCG adds a 7–10× wall‑clock penalty — Linaro, 2025.
  • glibc CPU‑feature detection produces cross‑runner Ir drift even on the same image — CodSpeed, 2026.

Score formula

score = 100 × meanworkloads( 0.55·e−CVwall/0.03 + 0.25·e−CVIr/0.002 + 0.20·e−(p99/med − 1)/0.20 )

Weighted toward wall‑clock CV because that is what a regression gate directly measures. Ir CV surfaces subtle environment drift (glibc, CPU features). The tail term penalizes bimodal distributions that ambush 2σ thresholds.

Data provenance

Sample values in this dashboard are synthetic but calibrated to published benchmark variance reported by CodSpeed, MongoDB, Linaro, and RunsOn. The generator (generate_data.py) emits data.json in the same shape a real harness would produce, so a real benchmark run can drop in without dashboard changes.
